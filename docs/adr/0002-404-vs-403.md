# 0002 — 404 for cross-workspace resources, 403 for visible-but-forbidden

**Status:** Accepted

## Context

Folio is multi-tenant: a member belongs to exactly one workspace. When a caller requests
a resource they may not access, the status code itself can leak information — a `403`
confirms the resource *exists*, which across a tenant boundary is a small data leak.

## Decision

`PageAuthorization` distinguishes two denial modes (`AccessResult.NotFound` vs
`Forbidden`), which services map to HTTP codes:

- **Different workspace → 404.** A resource in another workspace is reported as *not
  found* so its existence is indistinguishable from a truly missing id.
- **Private page, non-Owner → 403.** Within your own workspace, a page you can see to
  exist but may not access returns *forbidden*.
- **Insufficient write permission → 403** (Viewer, or Editor without `Edit`).

The same rule filters *listings*: `CanSeeVisibility` removes private pages from trees,
search, favorites, and backlinks for non-Owners, so they never appear at all.

## Consequences

- Cross-tenant probing is defeated: foreign ids and missing ids look identical.
- Clients must treat `404` on a known id as "gone or not yours" and `403` as "exists but
  blocked". The API reference documents this per endpoint.
- Tested exhaustively by `PermissionMatrixTests` (the role × visibility × permission
  matrix and foreign-workspace 404s across every read endpoint).
